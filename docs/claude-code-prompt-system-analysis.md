# Claude Code 提示词系统技术分析

## 概述

Claude Code 提示词系统是一个复杂的多层架构，旨在为 Claude API 动态构建、管理和优化提示词。它采用模块化、基于分节的方法，结合智能缓存、特性门控和上下文感知的提示词组装。

---

## 1. 架构设计

### 1.1 核心架构原则

提示词系统遵循以下关键架构原则：

| 原则 | 实现 |
|-----------|----------------|
| **模块化** | 提示词由离散的、可重用的分节组成 |
| **惰性求值** | 分节仅在需要时才计算 |
| **缓存策略** | 多级缓存，支持全局和会话范围的缓存键 |
| **特性门控** | 对特性标记的内容进行死代码消除 (DCE) |
| **类型安全** | 使用品牌类型进行编译时提示词验证 |

### 1.3 核心组件

#### SystemPrompt (`src/utils/systemPrompt.ts`)
基于优先级构建有效系统提示词的中央编排器：

```typescript
优先级（从高到低）：
1. 覆盖系统提示词（循环模式 - 替换所有内容）
2. 协调器系统提示词（协调器模式）
3. Agent 系统提示词（自定义 Agent）
4. 自定义系统提示词（--system-prompt CLI 标志）
5. 默认系统提示词（标准 Claude Code）
```

#### SystemPromptType (`src/utils/systemPromptType.ts`)
用于类型安全的品牌类型系统：

```typescript
export type SystemPrompt = readonly string[] & {
  readonly __brand: 'SystemPrompt'
}

export function asSystemPrompt(value: readonly string[]): SystemPrompt {
  return value as SystemPrompt
}
```

#### SystemPromptSections (`src/constants/systemPromptSections.ts`)
支持缓存的分节管理：

```typescript
type SystemPromptSection = {
  name: string
  compute: ComputeFn
  cacheBreak: boolean  // 控制缓存失效
}

// 记忆化分节 - 计算一次，缓存
function systemPromptSection(name, compute)

// 易失性分节 - 每轮重新计算
function DANGEROUS_uncachedSystemPromptSection(name, compute, reason)
```

## 2. 分层架构

### 2.1 三层提示词栈

提示词系统被组织成三个不同的层：

```
┌─────────────────────────────────────────────────────────────────┐
│ 第 1 层：静态前缀（全局缓存范围）                                   │
│ ─────────────────────────────────────────                      │
│ • 身份介绍                                                      │
│ • 核心系统指令                                                  │
│ • 工具使用指南（静态部分）                                        │
│ • 编码标准                                                      │
│ • 语气和风格规则                                                 │
│                                                                 │
│ 缓存键：跨组织共享（内容的 Blake2b 哈希）                          │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│ 第 2 层：动态边界                                                │
│ ─────────────────────────                                        │
│ 标记：__SYSTEM_PROMPT_DYNAMIC_BOUNDARY__                        │
│                                                                 │
│ 目的：将可缓存的静态内容与                                        │
│ 会话特定的动态内容分开                                           │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│ 第 3 层：动态内容（会话缓存范围）                                   │
│ ─────────────────────────────────────────────                    │
│ • 会话特定指南                                                  │
│ • 环境信息（当前工作目录、git 状态等）                              │
│ • MCP 服务器指令                                                │
│ • 记忆提示词                                                    │
│ • 语言偏好                                                      │
│ • 输出样式配置                                                  │
│ • 特性标记的分节                                                 │
│                                                                 │
│ 缓存键：会话特定                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 2.2 分节类别

#### 身份与介绍分节
- `getSimpleIntroSection()` - 核心身份声明
- `getCLISyspromptPrefix()` - CLI 归属前缀
- `CYBER_RISK_INSTRUCTION` - 安全指南

#### 系统指令分节
- `getSimpleSystemSection()` - 核心系统行为
- `getSystemRemindersSection()` - 系统提醒标签说明
- `getHooksSection()` - 用户钩子配置

#### 任务执行分节
- `getSimpleDoingTasksSection()` - 软件工程任务指南
- `getActionsSection()` - 操作安全性和可逆性
- `getUsingYourToolsSection()` - 工具选择指南

#### 工具特定分节
每个工具都有自己的提示词文件（例如 `src/tools/BashTool/prompt.ts`）：
- 工具描述
- 使用说明
- 安全指南
- 示例

## 5. 关键技术与实现细节

### 5.1 死代码消除 (DCE)

系统使用 Bun 的 bundle 时特性标记进行 DCE：

```typescript
import { feature } from 'bun:bundle'

// 条件导入 - 如果特性为 false，整个模块被消除
const proactiveModule =
  feature('PROACTIVE') || feature('KAIROS')
    ? require('../proactive/index.js')
    : null

// 条件代码块在构建时被消除
if (feature('TOKEN_BUDGET')) {
  // 此代码从外部构建中完全移除
}
```

### 5.2 缓存策略

#### 双级缓存系统

```typescript
// 第 1 层：系统提示词分节缓存
// - 每会话内存缓存
// - 跨轮次保留
// - 在 /clear 或 /compact 时清除

const cache = getSystemPromptSectionCache()
if (!section.cacheBreak && cache.has(section.name)) {
  return cache.get(section.name)
}

// 第 2 层：API 级提示词缓存
// - 全局范围：跨组织可缓存的前缀
// - 会话范围：用户特定内容
```

#### 缓存中断检测

```typescript
// src/services/api/promptCacheBreakDetection.ts

// 监控使提示词缓存失效的更改：
1. MCP 服务器连接/断开
2. Agent 定义更改
3. 权限模式更改
4. 设置更改

// 自动缓存失效时机：
- 允许列表上的文件更改
```

| **全局缓存范围** | 跨组织提示词前缀共享 | `splitSysPromptPrefix()` |
| **分节记忆化** | 避免重新计算静态分节 | `systemPromptSection()` |
| **Agent 列表附件** | ~10.2% cache_creation token 减少 | `shouldInjectAgentListInMessages()` |
| **MCP 增量附件** | 消除每轮 MCP 重新计算 | `isMcpInstructionsDeltaEnabled()` |
| **构建时 DCE** | 完全消除特性门控代码 | `feature()` 来自 `bun:bundle` |
| **Blake2b 缓存键** | 快速、确定性的缓存键生成 | API 层 |

## 9. 安全考量

1. **CYBER_RISK_INSTRUCTION**：内置于所有提示词路径
2. **沙箱感知**：工具提示词包含沙箱限制
3. **归属剥离**：隐藏模式防止数据泄露
4. **URL 生成限制**：模型不能生成任意 URL
5. **权限边界**：系统提示词强制执行权限模式语义

## 10. 总结

Claude Code 提示词系统代表了一种生产级的提示词工程方法，具有以下特点：

- **模块化设计**：可组合的分节，边界清晰
- **智能缓存**：多级缓存，支持全局/会话范围
- **特性灵活性**：构建时和运行时特性门控
- **类型安全**：品牌类型防止意外误用
- **性能优先**：针对 token 效率和缓存命中优化
- **可扩展性**：易于添加新分节、工具和模式

该架构成功平衡了灵活性（支持多种模式，如主动、协调器和基于 Agent 的工作流）与效率（积极的缓存和 DCE 优化）。
