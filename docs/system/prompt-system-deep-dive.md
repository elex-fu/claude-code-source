# Prompt System 深度技术思考

## 一、Prompt 作为系统界面

### 1.1 双向界面的不对称性

在 Claude Code 中，Prompt 不仅是「给模型的输入」，更是「系统与模型之间的契约」。这种双向性带来了独特的设计挑战：

**人类可读 vs 机器优化**
- CLAUDE.md 是人类编写的，需要清晰、结构化、易维护
- System Prompt 是给模型消费的，需要精确、紧凑、利于缓存

系统通过「分层注入」解决这一矛盾：
```
CLAUDE.md（人类编写）
    │
    ▼ 解析与转换
Context Assembly
    │
    ▼ 缓存优化
System Prompt Blocks（带 cache_control 标记）
    │
    ▼ API 传输
Anthropic API
```

### 1.2 缓存作为架构原则

Prompt Caching 不是事后优化的附加功能，而是深刻影响了架构设计的核心策略：

**物理边界的设计**
`SYSTEM_PROMPT_DYNAMIC_BOUNDARY` 是一个物理切分点，它强制要求：
- 边界之前的内容是静态的（跨会话复用）
- 边界之后的内容是动态的（每会话变化）

这种硬性边界使得缓存策略可以被「编译」到 prompt 结构中，而非运行时动态判断。

**缓存作用域的层级**
```
global ──▶ 所有用户共享（核心指令）
org    ──▶ 企业内共享（企业规则）
null   ──▶ 不缓存（用户特定上下文）
```

这种层级设计体现了「越通用的内容，缓存收益越大」的经济学原理。

## 二、消息系统的信息论视角

### 2.1 消息作为熵的载体

从信息论角度，对话历史是一个熵不断增加的系统：
- 每一轮对话引入新的信息（用户输入、工具结果）
- 模型的思考过程产生冗余（redacted_thinking）
- 附件和元数据增加噪声

Prompt System 的核心任务之一是「熵管理」：
- **Compact**：低熵摘要替换高熵原始数据
- **Snip**：删除过期的高熵内容
- **Boundary**：明确标记熵的变化点

### 2.2 消息 ID 的拓扑稳定性

`deriveUUID` 函数确保了消息拆分时的 ID 稳定性：

```
原始消息 M ──▶ 拆分为 M1, M2, M3
                 │
                 ▼
          ID(M1) = f(ID(M), 1)
          ID(M2) = f(ID(M), 2)
          ID(M3) = f(ID(M), 3)
```

这种确定性映射保证了：
- 相同的历史，规范化后得到相同的 ID 序列
- React 的 reconciliation 不会因 ID 抖动而产生不必要重渲染
- 附件和进度条能稳定绑定到对应内容块

关键洞察：**ID 不仅是标识符，更是消息「血统」的编码**。

## 三、上下文压缩的策略博弈

### 3.1 压缩作为有损编码

Auto-Compact 本质上是一个有损压缩算法：
- **输入**：高维的对话历史（包含代码片段、工具输出、思考过程）
- **输出**：低维的摘要（自然语言描述）
- **损失**：具体代码行、精确的错误信息、时序细节

系统的创新在于「可控的损失」：
- 通过 `postCompactCleanup` 恢复最多 5 个关键文件
- 保留最近的 tool_use/tool_result 对（模型注意力焦点）
- 用户可以通过 `@file` 显式保留重要内容

### 3.2 压缩触发器的博弈论

何时触发压缩是一个优化问题：
- **过早压缩**：损失有价值信息，影响模型效果
- **过晚压缩**：面临 API 的 hard limit（413 错误），被迫更激进压缩

系统采用「安全缓冲」策略：
```
触发阈值 = 模型上下文窗口 - 安全缓冲(13k tokens)
```

这种保守策略确保了：
- 总有足够空间给模型生成回复
- 压缩在「被迫中断」之前主动发生，体验更平滑

## 四、系统提示词的认知架构

### 4.1 提示词的心智模型构建

Claude Code 的系统提示词不仅是「指令集」，更是在构建一个「心智模型」：

**角色定位**
- 「你是一个终端 AI 助手」—— 明确角色边界
- 「你应该谨慎对待文件操作」—— 建立风险意识

**工作记忆管理**
- 「旧工具结果将被自动清除」—— 告知上下文限制
- 「你可以通过 @file 引用文件」—— 教授外部记忆策略

**元认知提示**
- 「当你不确定时，请询问用户」—— 校准置信度
- 「你应该验证你的假设」—— 鼓励自我修正

这种提示词设计借鉴了认知心理学中的「元认知」概念——不仅告诉模型做什么，还告诉它如何思考。

### 4.2 工具描述的界面隐喻

工具描述不是简单的函数签名，而是「 affordance 提示」：

```
Bash: "执行 shell 命令...仅当你需要运行计算或启动开发服务器时"
     │
     └── 明确的使用场景引导

FileEdit: "编辑文件...使用此工具进行集中的 surgical 替换"
         │
         └── 强调使用时机和方式
```

这些描述引导模型形成正确的「工具使用直觉」，减少误用。

## 五、规范化的编译器思维

### 5.1 多 Pass 清洗的编译器模式

`normalizeMessagesForAPI` 是一个典型的「编译器 Pipeline」：

```
Source Messages
       │
       ▼
Pass 1: 附件重排（attachments first）
       │
       ▼
Pass 2: 工具引用修复（ensureToolResultPairing）
       │
       ▼
Pass 3: 图片大小检查与容错
       │
       ▼
Pass 4: 消息合并（连续 user/assistant 消息）
       │
       ▼
Pass 5: Orphan thinking 过滤
       │
       ▼
Target API Messages
```

每个 Pass 解决特定问题，Pass 之间有明确的依赖顺序。这种设计虽然增加了复杂性，但使得：
- 每个 Pass 可以独立测试
- 新 Pass 可以安全插入
- 问题可以定位到具体 Pass

### 5.2 错误恢复的优雅降级

规范化过程不追求「全有或全无」，而是「尽最大努力」：
- 图片过大？缩放或返回错误占位符，而不是整个请求失败
- Tool result 配对失败？插入合成占位符，保持结构完整
- 消息格式异常？剥离异常部分，保留有效内容

这种「容错规范化」确保了即使上游（模型、工具）产生异常输出，系统仍能生成合法的 API 请求。

## 六、反思与展望

### 6.1 当前架构的权衡

**多 Pass 的脆弱性**
当前 10+ 个 Pass 的顺序是经验性的，某些 Pass 之间存在隐式依赖。未来可能需要重构为数据流图（Dataflow Graph），显式声明依赖关系。

**缓存粒度的两难**
过大的缓存块（global）收益高但不灵活；过小的缓存块（per-message）灵活但收益低。当前的三层（global/org/null）是工程妥协，但可能不是最优。

### 6.2 未来演进方向

**语义化缓存**
基于内容的哈希而非物理位置进行缓存。例如，两个不同的 CLAUDE.md 文件如果语义等价（通过 embedding 判断），可以共享缓存。

**自适应提示词**
根据对话历史动态调整 system prompt。例如，在长会话中逐渐强化「请引用先前信息」的提示，在代码密集型对话中增加编程特定的指导。

**多模态提示词的统一**
当前图片、文本、工具结果有各自的规范化路径。未来可以探索统一的「富媒体消息」抽象，简化架构。

---

Claude Code 的 Prompt System 展示了如何将「提示工程」提升为「提示架构」——它不仅是写好 prompt 的艺术，更是构建稳定、可缓存、可维护的 LLM 交互基础设施的科学。理解其设计，对于任何需要与 LLM 深度集成的系统都具有重要参考价值。
