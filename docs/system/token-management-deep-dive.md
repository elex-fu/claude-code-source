# Token Management 深度技术思考

## 一、Token 作为系统资源

### 1.1 从计费单位到架构约束

在 LLM 应用中，Token 首先是计费单位，但在 Claude Code 的架构中，Token 已经上升为一种「系统资源」，类似于 CPU、内存、磁盘空间：

```
┌─────────────────────────────────────────────────────┐
│  CPU ──▶ 计算密集型任务调度                           │
│  Memory ──▶ 应用状态缓存                              │
│  Disk ──▶ 持久化存储                                  │
│  Token ──▶ 上下文窗口管理与 API 调用预算              │
└─────────────────────────────────────────────────────┘
```

这种认知转变带来了架构设计的根本变化：
- **预算感**：系统时刻感知 Token 的「余额」和「消费速率」
- **调度策略**：当资源紧张时，有明确的优先级和降级策略
- **监控告警**：成本跟踪、预警、限制

### 1.2 Token 的流动性与压力

Token 在 Claude Code 中不是静态的，而是在持续流动：

```
用户输入 ──▶ 输入 Token 消耗
    │
    ▼
模型处理 ──▶ 计算 Token 消耗
    │
    ▼
模型输出 ──▶ 输出 Token 消耗
    │
    ▼
工具结果 ──▶ 结果 Token 加入上下文
    │
    ▼
Compact ──▶ Token 回收（通过摘要压缩）
```

这种流动性要求系统具备「实时压力感知」能力——不是事后统计，而是预测和干预。

## 二、成本追踪的工程实现

### 2.1 多维度成本归因

Claude Code 的成本追踪不是简单的「总花费」，而是细粒度的多维归因：

```
总成本
├── 按模型拆分
│   ├── Claude Opus ──▶ 高成本，用于复杂任务
│   ├── Claude Sonnet ──▶ 中成本，默认选择
│   └── Claude Haiku ──▶ 低成本，用于简单任务
│
├── 按功能拆分
│   ├── 主 Query ──▶ 核心对话成本
│   ├── Compact ──▶ 上下文压缩成本
│   ├── Tool Use Summary ──▶ 结果摘要成本
│   └── Agent Subtask ──▶ 子代理成本
│
└── 按会话拆分
    ├── 当前会话
    └── 历史会话（用于趋势分析）
```

这种归因使得：
- **优化有方向**：知道哪里花钱最多，就知道优化哪里
- **预算可分配**：可以为不同功能设置子预算
- **决策有依据**：选择模型时有数据支持

### 2.2 实时成本估算的博弈

成本追踪面临一个挑战：API 返回的 usage 是事后统计，而决策（如是否触发 compact）需要事前预测。

**解决方案：双层估算**
```
粗略估算（本地）
├── 基于字符数 * 语言系数
├── 快速、低成本、准确度中等
└── 用于实时决策（如是否触发 compact）

精确统计（API 返回）
├── 基于 tokenizer 的实际计算
├── 准确、事后、有延迟
└── 用于最终计费和报表
```

这种「快-慢」双系统架构确保了：
- 实时决策有足够快的输入
- 最终统计有足够高的精度

## 三、上下文压缩的策略博弈论

### 3.1 压缩作为资源置换

Auto-Compact 本质上是一个「资源置换」策略：

```
原始上下文 ──▶ 高 Token 成本，高信息保真度
    │
    ▼ 压缩
摘要上下文 ──▶ 低 Token 成本，低信息保真度
```

关键问题：**何时置换？置换多少？**

Claude Code 的策略是基于「安全边际」：
```
触发压缩 = 当前 Token > (上下文窗口 - 安全缓冲)
```

安全缓冲（13k tokens）是留给：
- 模型生成回复的空间
- 工具调用所需的上下文
- 意外 Token 增长的余量

### 3.2 四级压缩的渐进降级

四级压缩策略体现了「最小必要损失」原则：

```
Level 1: Snip Compact
├── 损失：冗余的思考过程、已处理的工具输出
└── 收益：几乎零成本回收

Level 2: Micro Compact
├── 损失：单轮内的详细工具链
└── 收益：保持多轮历史，仅压缩当前轮

Level 3: Auto Compact
├── 损失：老旧对话的具体内容
└── 收益：保留长期记忆（摘要形式）

Level 4: Context Collapse
├── 损失：多轮对话结构
└── 收益：极端情况下的生存
```

每一级都是前一级的安全网，确保系统在任何情况下都不会「硬着陆」（API 413 错误）。

### 3.3 压缩的副作用管理

压缩不是免费的，它有副作用：
- **延迟**：fork agent 生成摘要需要时间
- **成本**：摘要本身消耗 Token
- **信息损失**：细节被抽象为概要

Claude Code 通过「后压缩恢复」缓解信息损失：
```
压缩后 ──▶ 识别摘要中引用的关键文件
    │
    ▼
重新加载这些文件（预算允许范围内）
    │
    ▼
注入上下文 ──▶ 恢复关键细节
```

这种「先压缩后恢复」的策略，在「保持上下文健康」和「保留关键信息」之间取得平衡。

## 四、预算控制的反馈循环

### 4.1 Token 预算的反馈机制

Claude Code 实现了「感知-决策-执行」的反馈循环：

```
感知：tokenCountWithEstimation()
           │
           ▼
决策：calculateTokenWarningState()
           │
           ├── safe ──▶ 正常执行
           ├── warning ──▶ 提示用户
           └── critical ──▶ 强制 compact
           │
           ▼
执行：compact / blocking / continue
```

这种闭环确保了系统不会「失控」——即使面对无限长的对话，也能在资源耗尽前采取行动。

### 4.2 用户可见的成本透明度

`/cost` 命令提供了完全透明的成本视图：

```
当前会话成本
├── Token 使用量（输入/输出/缓存）
├── 按模型细分
├── 与历史会话对比
└── 预估剩余预算
```

这种透明性有重要的心理学意义：
- **信任**：用户知道钱花在哪里
- **控制**：用户可以基于成本做决策
- **学习**：用户逐渐理解不同操作的成本

## 五、缓存的经济学

### 5.1 Prompt Caching 的收益模型

Prompt Caching 是 Claude Code 成本优化的核心：

```
无缓存：
每次请求 ──▶ 完整上下文 ──▶ 高成本

有缓存：
首次请求 ──▶ 完整上下文 ──▶ 高成本（写入缓存）
后续请求 ──▶ 缓存命中 ──▶ 低成本（仅读缓存）
```

缓存的经济学依赖于「写入一次，读取多次」的模式：
- System Prompt：写入 1 次，每会话读取 N 次
- 长对话历史：增量写入，整体读取

### 5.2 缓存失效的策略

缓存不是银弹，它需要失效策略：

```
时间失效：长时间未使用，缓存过期
内容失效：内容变化，缓存标记为 dirty
手动失效：CACHE_BREAKER 强制刷新
```

Claude Code 的 `SYSTEM_PROMPT_DYNAMIC_BOUNDARY` 是一种「结构化失效」——只使变动部分失效，保留静态部分。

## 六、反思与演进

### 6.1 当前架构的权衡

**估算 vs 精确的永恒张力**
粗略估算用于实时决策，但不准确；精确统计用于计费，但有延迟。这种张力无法完全消除，只能管理。

**压缩的不可逆性**
一旦压缩，原始信息就丢失了（除非有持久化）。当前系统通过「附件」和「文件引用」部分缓解了这个问题，但完美恢复是不可能的。

### 6.2 未来演进方向

**预测性压缩**
基于对话趋势预测 Token 增长，提前启动压缩，避免「紧急压缩」造成的用户体验抖动。

**自适应安全缓冲**
不同模型、不同场景需要不同的安全缓冲。可以基于历史数据动态调整缓冲大小。

**Token 预算的游戏化**
让用户设置预算目标，系统提供「省钱建议」（如「使用 Sonnet 而非 Opus 可节省 50%」）。

---

Claude Code 的 Token Management 是一个将「成本控制」上升为「系统架构」的典范。它不仅追踪花费，更通过预算控制、压缩策略、缓存优化等手段，确保了系统的可持续运行。理解其设计，对于构建任何生产级 LLM 应用都具有重要参考价值。
