# Multi-Agent Coordination 深度技术思考

## 一、并行智能体的哲学基础

### 1.1 从串行到并行的范式转移

传统 LLM 应用采用「单智能体-单会话」模型，Claude Code 的多智能体协调实现了根本性的范式转移：

```
传统模型：                多智能体模型：
┌─────────┐              ┌─────────────┐
│  User   │              │    User     │
└────┬────┘              └──────┬──────┘
     │                          │
     ▼                          ▼
┌─────────┐              ┌─────────────┐
│  Agent  │              │ Coordinator │
│ (唯一)  │              │  (协调者)    │
└────┬────┘              └──────┬──────┘
     │                          │
     ▼                   ┌──────┴──────┐
┌─────────┐              ▼             ▼
│ Result  │         ┌────────┐    ┌────────┐
└─────────┘         │Agent A │    │Agent B │
                    └───┬────┘    └───┬────┘
                        │             │
                        └──────┬──────┘
                               ▼
                         ┌─────────┐
                         │ Combined│
                         │ Result  │
                         └─────────┘
```

这种转移的核心洞察是：**复杂问题天然可分解**，而 LLM 的确定性延迟使得并行成为加速的关键。

### 1.2 三种隔离级别的工程权衡

Claude Code 提供了三种智能体隔离机制，体现了「安全-性能」权衡的连续谱：

| 隔离级别 | 实现机制 | 启动延迟 | 内存开销 | 适用场景 |
|---------|---------|---------|---------|---------|
| 进程隔离 (fork) | 子进程 + 独立内存 | 秒级 | 300MB+ | 高安全性要求 |
| 环境隔离 (env) | 环境变量标识 | 毫秒级 | 共享 | 多终端协作 |
| 上下文隔离 (AsyncLocalStorage) | 异步上下文存储 | 微秒级 | 共享 | 进程内并发 |

关键在于：**不追求绝对隔离，而追求「足够好的隔离」**。进程内 AsyncLocalStorage 方案使得创建同伴智能体的开销接近于零，这让「智能体即函数」的编程模型成为可能。

## 二、身份与上下文的分布式一致性

### 2.1 AsyncLocalStorage 的魔力

Node.js 的 `AsyncLocalStorage` 是一个被低估的原语。在多智能体场景中，它解决了「谁在执行」的根本问题：

```
主线程
  │
  ├── runWithTeammateContext(ctxA, fnA)
  │     │
  │     ├── Promise 1 (属于 Agent A)
  │     │     └── getTeammateContext() → ctxA ✓
  │     │
  │     └── Promise 2 (属于 Agent A)
  │           └── getTeammateContext() → ctxA ✓
  │
  └── runWithTeammateContext(ctxB, fnB)
        │
        ├── Promise 3 (属于 Agent B)
        │     └── getTeammateContext() → ctxB ✓
        │
        └── Promise 4 (属于 Agent B)
              └── getTeammateContext() → ctxB ✓
```

无论 Promise 如何交织执行，`getTeammateContext()` 总能返回正确的上下文。这种「隐式参数传递」消除了显式 context 参数的传播负担。

### 2.2 三层身份解析的韧性设计

```typescript
getAgentId() 的解析优先级：

1. AsyncLocalStorage (最精确)
   └── 仅存在于进程内智能体执行期间
   
2. DynamicTeamContext (动态加入)
   └── 运行时加入团队的普通进程
   
3. 环境变量 (最通用)
   └── CLAUDE_CODE_AGENT_ID
   └── 适用于 tmux/iTerm2 等多终端场景
```

这种层级设计确保了「优雅降级」——即使高级机制失效，系统仍能基于环境变量正确识别身份。

## 三、Mailbox：文件系统的消息隐喻

### 3.1 为何选择文件系统？

在多智能体通信方案的选择上，Claude Code 选择了「文件系统邮箱」这一看似朴素的技术：

```
~/.claude/teams/{team}/inboxes/{agent}.json
```

这种选择的深层原因：

1. **持久性**：消息不因进程崩溃而丢失
2. **可观测性**：开发者可以直接 `cat` 查看消息内容
3. **跨进程**：不受语言运行时限制，任何进程都能读写
4. **原子性**：配合 lockfile 实现可靠的原子操作
5. **无需额外依赖**：不依赖 Redis、RabbitMQ 等外部服务

### 3.2 竞态解决的乐观锁模式

```
写入方                          读取方
  │                               │
  ├── 1. 获取 lockfile ─────────▶ │
  │                               │
  ├── 2. 读取 mailbox.json       │
  │   (获取最新状态)               │
  │                               │
  ├── 3. 追加新消息              │
  │                               │
  ├── 4. 写入 mailbox.json       │
  │                               │
  ├── 5. 释放 lockfile ─────────▶ │
  │                               │
  │                               ├── 6. 获取 lockfile
  │                               │
  │                               ├── 7. 读取 mailbox.json
  │                               │   (看到新消息)
  │                               │
  │                               └── 8. 释放 lockfile
```

`proper-lockfile` 库提供的「stale 检测」机制（5秒超时自动释放）防止了死锁，即使写入方崩溃也不会永久阻塞。

### 3.3 消息类型的语义设计

Mailbox 支持的消息类型体现了分布式系统的经典模式：

- **权限请求/响应**：两阶段提交的简化版
- **关闭请求/确认**：优雅关闭的握手协议
- **计划审批请求/响应**：工作流编排的同步点

这些类型不是随意定义的，而是对应了多智能体协作中的「关键决策点」。

## 四、任务框架：统一的多态抽象

### 4.1 任务作为一等公民

Claude Code 将「执行中的智能体」抽象为「任务」，这一抽象统一了多种执行模式：

```typescript
type TaskState =
  | BashTaskState           // shell 命令
  | InProcessTeammateTaskState  // 进程内同伴
  | LocalAgentTaskState     // 本地异步智能体
  | RemoteAgentTaskState    // 远程会话智能体
  | DreamTaskState          // 规划/反思智能体
```

这种统一使得 UI 可以用同一套组件渲染所有任务状态，调度器可以用同一套逻辑管理任务生命周期。

### 4.2 状态机的幂等设计

```
         ┌─────────┐
         │ initial │
         └────┬────┘
              │ 启动
              ▼
         ┌─────────┐
    ┌───▶│ running │◀──┐
    │    └────┬────┘   │
    │         │        │ 重试
完成│    错误  │        │
    │         ▼        │
    │    ┌─────────┐   │
    └───┤│  failed │───┘
         └────┬────┘
              │ 杀死
              ▼
         ┌─────────┐
         │  killed │
         └─────────┘
```

任务状态转换被设计为「幂等」的——多次执行同一转换不会产生副作用。这在分布式环境中至关重要，因为网络重试可能导致重复指令。

## 五、协调者模式：分治的艺术

### 5.1 领导者的责任边界

在 Coordinator Mode 中，主智能体（Coordinator）承担以下责任：

1. **任务分解**：将复杂问题拆分为可并行子任务
2. **智能体选择**：为每个子任务匹配合适的 Agent 定义
3. **权限桥接**：将同伴的权限请求转发给用户
4. **结果聚合**：汇总多个智能体的输出
5. **冲突解决**：处理智能体间的资源竞争

### 5.2 权限桥接的信任链

```
同伴智能体请求权限
         │
         ▼
┌─────────────────┐
│  bridgeTeammate │  创建 Promise，注册到
│  PermissionTo   │  teammatePermissionQueue
│    Leader()     │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│  领导者 UI 显示  │  PermissionRequestDialog
│  权限请求弹窗    │
└────────┬────────┘
         │
         ├── 用户批准 ──▶ 通过 Mailbox 发送响应
         │                    │
         │                    ▼
         │            ┌─────────────────┐
         │            │ 同伴 Promise    │
         │            │    resolve()    │
         │            └─────────────────┘
         │
         └── 用户拒绝 ──▶ 同上，但拒绝
```

这种设计让同伴智能体「感知不到」自己是在被代理——它只是在等待一个普通的权限响应。

## 六、资源治理的隐性智慧

### 6.1 消息历史的容量控制

为防止内存无限增长，同伴智能体的消息历史有 50 条上限：

```
当 messages.length > 50:
  1. 保留所有 system 消息
  2. 按时间排序非 system 消息
  3. 移除最早的多余消息
```

这一设计承认了「远因遗忘」的认知规律——太久之前的对话细节对当前任务的帮助有限，而 system 消息（如身份定义）始终重要。

### 6.2 优雅关闭的契约精神

```
领导者                        同伴智能体
  │                               │
  ├── shutdown_request ─────────▶ │
  │   (Mailbox)                   │
  │                               │
  │                               ├── 完成当前工具调用
  │                               ├── 清理资源
  │                               ├── 发送最终状态更新
  │                               └── 自然退出
  │                               │
  │◀──────── 超时 ────────────────┤
  │   (5秒后)                     │
  │                               │
  ├── AbortController.abort() ──▶ │
  │   (强制终止)                   │
```

这种「先礼后兵」的关闭策略确保智能体有机会保存进度，而非被强制杀死。

## 七、反思与演进

### 7.1 当前架构的权衡

**Mailbox 的吞吐量限制**

基于文件的通信在极高频场景（每秒数百条消息）下会成为瓶颈。但在实际使用场景中，智能体间的通信频率远低于此，因此简单性胜过性能。

**进程内隔离的边界**

AsyncLocalStorage 虽然高效，但它无法隔离：
- 全局变量修改
- 文件系统副作用
- 网络连接

这些需要依赖「信任」——相信智能体不会恶意破坏。

### 7.2 未来演进方向

**分布式团队**

当前团队机制局限于单机。未来可通过网络传输 Mailbox 消息，实现真正的分布式多机协作。

**动态能力发现**

智能体可以注册自己的能力（「我擅长代码审查」、「我能运行测试」），协调者据此动态分配任务，而非依赖预定义配置。

**智能体间直接通信**

当前通信必须通过 Mailbox 间接进行。未来可支持 P2P 直接消息，用于需要低延迟协作的场景（如配对编程）。

---

Claude Code 的多智能体协调模块是一个将「分布式系统原理」应用于 AI 协作的典范。它通过 AsyncLocalStorage 实现了零开销的进程内隔离，通过 Mailbox 实现了可靠的跨进程通信，通过任务框架统一了多态执行模型。理解其设计，对于构建任何需要并行智能体协作的系统都具有重要参考价值。
