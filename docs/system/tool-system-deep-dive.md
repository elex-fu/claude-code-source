# Tool System 深度技术思考

## 一、安全模型的设计哲学

### 1.1 Fail-Closed 与显式授权

Claude Code 工具系统的默认姿态是「拒绝」而非「允许」。这种 Fail-Closed 设计体现在多个层面：

**Schema 层面的拒绝**
所有工具输入必须通过 Zod Schema 验证。未定义的字段、类型不匹配、缺失必填项都会导致验证失败。这与传统 Web 应用的「宽松解析」哲学相反——在 AI 代理场景中，「猜测用户意图」是危险的，因为 LLM 可能产生幻觉输入。

**并发安全层面的拒绝**
`isConcurrencySafe` 默认返回 `false`。工具开发者必须显式证明其操作是安全的（无副作用、无竞态条件）才能开启并发。这种设计避免了「默认并行导致的数据竞争」陷阱。

**权限层面的拒绝**
权限规则中，`deny` 列表优先于 `allow` 列表。即使某个操作匹配了允许规则，只要它同时匹配了拒绝规则，就会被阻止。这种「黑名单优先」策略确保企业管理员可以通过简单的规则覆盖用户配置。

### 1.2 权限的三层架构

工具权限不是单一检查点，而是一个分层防御体系：

```
第一层：静态规则匹配
├── alwaysDenyRules  （最高优先级）
├── alwaysAllowRules （仅当无 deny 匹配时）
└── alwaysAskRules   （强制交互确认）

第二层：动态模式判断
├── auto-mode 分类器（YoloClassifier）
├── plan-mode 拦截器
└── accept-mode 放行器

第三层：运行时交互
├── 权限对话框
├── 自动超时处理
└── 决策缓存（避免重复询问）
```

这种分层使得权限策略可以从「完全自动化」平滑过渡到「完全人工控制」，适应从个人开发到企业审计的不同场景。

## 二、工具抽象的通用性设计

### 2.1 Tool 类型的代数结构

`Tool<Input, Output, P>` 的泛型设计体现了一种代数思维：

- **Input** 是输入空间的类型，由 Zod Schema 定义
- **Output** 是输出空间的类型，决定 tool_result 的结构
- **P** 是进度空间的类型，允许工具流式报告中间状态

这种参数化使得工具系统可以处理异构的工具集合（Bash 输出字符串、Agent 输出消息流、MCP 输出 ContentBlocks），同时保持类型安全。

关键洞察：**进度（Progress）作为独立类型参数**，允许「只读」进度报告（如 Bash 实时输出）与「最终结果」分离。这对用户体验至关重要——用户希望看到命令实时输出，而不是等待最后一次性显示。

### 2.2 渲染的分离架构

工具的「逻辑执行」与「结果渲染」被刻意分离：

```
Tool.call() ──▶ 执行逻辑 ──▶ 返回结构化数据
                    │
                    ▼
            mapToolResultToToolResultBlockParam()
                    │
                    ▼
            渲染为 API 可用的 ContentBlock
                    │
                    ▼
            renderToolResultMessage() (可选)
                    │
                    ▼
            React 组件渲染（UI 层）
```

这种分离使得同一执行结果可以有不同的表示形式：
- API 看到的是简洁的 text/image 块
- UI 看到的是富交互组件（可展开的文件对比、带语法高亮的代码块）
- 日志看到的是纯文本审计轨迹

## 三、并发模型的工程权衡

### 3.1 并发分区的正确性证明

`partitionToolCalls` 的算法保证了：

**引理 1**：同一 batch 内的所有工具要么都是 concurrency-safe，要么都是 non-safe。
**引理 2**：non-safe batch 的大小为 1。
**引理 3**：batch 之间的执行顺序是串行的。

由此可推出**定理**：任意两个 non-safe 工具不会并发执行。

这种形式化保证虽然简单，但在工程实践中至关重要——它消除了竞态条件的根本来源。

### 3.2 流式与批量的统一

StreamingToolExecutor 和 toolOrchestration 看似是两个独立实现，但共享相同的核心抽象：

```typescript
// 共享的 MessageUpdate 类型
type MessageUpdate = {
  message?: Message          // 产生的消息（可能是进度）
  newContext?: ToolUseContext // 上下文修改
}
```

这种统一允许上层 Query Loop 以相同的方式消费两种执行模式的结果，无需关心底层是流式还是批量。

关键设计：**Context Modifier 模式**
工具执行过程中可以返回一个函数 `modifyContext`，在适当的时机（串行 batch 内立即应用，并发 batch 后批量应用）更新全局上下文。这种延迟应用避免了并发读写冲突。

## 四、MCP 集成的架构挑战

### 4.1 动态类型的处理

MCP 工具在连接前是未知的，这带来了 TypeScript 的类型挑战。解决方案是「运行时类型擦除 + 边界校验」：

```
MCP Tool Definition ──▶ 运行时加载
                              │
                              ▼
                      构建 Zod Schema（动态）
                              │
                              ▼
                      包装为 Tool<any, any, any>
                              │
                              ▼
                      在边界处（call 前后）进行类型校验
```

这种设计牺牲了部分编译时类型安全，换取了运行时的灵活性。类型校验被推迟到实际调用时，通过 Zod 保证。

### 4.2 结果大小的流量控制

MCP 工具可能返回巨大结果（如读取大文件、查询数据库）。系统采用「令牌桶」思想进行流量控制：

```
工具结果 ──▶ 大小检测
                │
    ┌──────────┴──────────┐
    ▼                     ▼
  小结果               大结果
    │                     │
    ▼                     ▼
  直接返回          持久化到磁盘
                          │
                          ▼
                    返回文件路径引用
```

这种「背压」机制防止单个工具调用耗尽上下文窗口，同时保留了数据的完整性（通过外部存储）。

## 五、Hooks 的插件架构

### 5.1 Pre/Post Tool Hooks 的拦截模型

工具执行前后可以插入自定义 hooks，形成 AOP（面向切面编程）风格的能力：

```
PreTool Hooks ──▶ 可以修改输入、阻止执行、提前返回结果
Tool Execution
PostTool Hooks ──▶ 可以修改输出、记录日志、触发副作用
```

这种设计允许用户在不修改工具源码的情况下：
- 添加审计日志
- 实现自定义权限检查
- 对结果进行后处理（如自动格式化代码）

### 5.2 Hooks 的副作用隔离

Hooks 运行在独立的 subprocess 中，与主进程隔离：
- Hook 崩溃不会影响主循环
- Hook 的资源消耗（CPU、内存）有独立限制
- Hook 可以安全地执行任意代码（如运行 linter、测试套件）

这种隔离性使得 Hooks 成为「用户自定义逻辑」的安全沙箱。

## 六、反思与演进

### 6.1 当前设计的边界

**工具组合的复杂性**
当前工具系统擅长处理「并行独立工具」和「串行依赖工具」，但对于「条件工具」（根据前一个结果决定下一个工具）的支持有限。这类逻辑目前需要在 Query Loop 中通过多轮对话实现，效率较低。

**长工具链的状态传递**
当工具链很长时（A → B → C → D），中间状态需要在多个 context modifier 中传递，调试变得困难。缺乏一个显式的「工作流状态」容器。

### 6.2 未来演进方向

**工作流原生支持**
可以引入显式的 Workflow 概念，允许用户定义工具之间的 DAG（有向无环图）依赖关系。系统可以基于此进行更智能的调度和可视化。

**工具的组合类型**
当前 `Tool<Input, Output>` 是单输入单输出。未来可以探索多输入（聚合多个工具结果）和多输出（一个工具触发多个后续动作）的类型设计。

**自适应并发**
当前并发度是固定的（默认 10）。可以基于系统负载、工具历史执行时间、网络状况动态调整并发度，实现更好的资源利用率。

---

Claude Code 的工具系统是一个经过精心打磨的工程成果，它在「安全性」与「灵活性」、「类型安全」与「动态扩展」、「性能」与「可预测性」之间取得了优雅的平衡。理解其设计哲学，对于构建任何需要与外部世界交互的 AI 系统都具有重要参考价值。
