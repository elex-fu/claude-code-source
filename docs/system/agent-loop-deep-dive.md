# Agent Loop 深度技术思考

## 一、架构设计哲学

### 1.1 生成器作为状态容器

Claude Code 的 Query Engine 采用 `AsyncGenerator` 作为核心抽象，这一选择蕴含深刻的架构思考：

**时间线的显式化**
传统 Promise/Callback 模型将异步操作隐藏在链式调用中，而 Generator 将时间切割为离散的 `yield` 点。每一个 `yield` 都是一个「可观测状态」，让外部系统（UI、日志、调试器）能够精确地介入和追踪。这种设计使得「暂停-恢复」成为一等公民能力，而非通过复杂的状态机模拟。

**背压的天然支持**
当 UI 渲染跟不上模型输出速度时，Generator 的 `next()` 调用天然形成背压。消费方不调用 `next()`，生产方就停留在 `yield` 处等待。这种反向压力传递避免了内存无限增长，同时保持代码的线性可读性。

**可测试性的提升**
Generator 可以被「冻结」在任意 yield 点，状态完全封装在局部变量中。测试代码可以：
- 迭代到特定步骤后注入 mock 数据
- 验证某一步是否正确 yield 了期望的事件类型
- 模拟中断场景（`.return()`）验证清理逻辑

### 1.2 分层的状态隔离

Query Loop 中存在三层状态：

| 层级 | 载体 | 生命周期 | 变更频率 |
|------|------|----------|----------|
| Immutable Params | 函数参数 | 整个 Query 调用 | 从不 |
| Mutable State | `let state` 对象 | 多轮 Turn | 每轮一次 |
| Ephemeral Context | `toolUseContext` 引用 | 单轮 Turn | 随时 |

这种分层不是随意为之，而是对应着不同的持久化需求：
- **Params** 是配置，应该被 memoize
- **State** 是检查点，需要支持序列化（用于 resume/session 恢复）
- **Context** 是运行时临时数据，可以丢失但重建成本高

关键洞察：`State` 的「伪不可变」更新（`state = { ...state, x: y }`）不是为了函数式编程的 purity，而是为了**快照能力**。每一轮的状态都可以被保存为一个完整的检查点，支持「时间旅行」调试。

## 二、流式执行的复杂性管理

### 2.1 流式工具执行的并发模型

StreamingToolExecutor 面临一个经典问题：
- 模型输出是**单线程顺序**的（token 逐个产生）
- 工具执行是**多线程并发**的（Bash、WebFetch 等 I/O 操作）
- UI 更新必须是**单线程顺序**的（React 渲染是同步的）

解决方案是一个「多生产者-单消费者」队列：
```
模型流 ──▶ [tool_use 检测] ──▶ 工具 A 开始执行 ──┐
                          └─▶ 工具 B 开始执行 ──┤
                                              ▼
                                          [结果队列]
                                              │
                                              ▼
                                          UI 渲染（单线程）
```

关键点：**结果的顺序性保证**。即使工具 B 先完成，如果它在消息序列中排在 A 后面，也要等待 A 完成后一起 yield。这种顺序性对 LLM 的推理至关重要——模型预期看到的是它发出 tool_use 的顺序对应的结果顺序。

### 2.2 Sibling Abort 的级联语义

`siblingAbortController` 实现了一种「同辈取消」语义：
- 当并行 Bash 命令中的一个失败时，取消其他正在运行的 Bash
- 但不取消并行的 FileRead 或 WebFetch

这种细粒度控制基于一个观察：**同类型工具往往有依赖关系，异类型工具通常是独立的**。

例如：
```bash
# 这些命令有依赖关系
cd /tmp && mkdir test && cd test
# 如果 mkdir 失败，cd 应该被取消

# 但这些操作是独立的
grep "pattern" file1.txt &  # 读操作
find . -name "*.js" &       # 另一个读操作
```

Sibling Abort 的设计哲学是「快速失败」——尽早停止无意义的工作，释放资源。

## 三、容错与恢复的策略层次

### 3.1 四级压缩的渐进降级

Query Loop 的上下文压缩策略体现了「渐进降级」的工程智慧：

```
Level 1: Reactive Compact  ──▶ 零成本，仅重新组织已有数据
Level 2: Micro Compact     ──▶ 低成本，单轮内摘要
Level 3: Auto Compact      ──▶ 中成本，fork 轻量 agent
Level 4: Context Collapse  ──▶ 高成本，激进摘要
```

每一级都是前一级的安全网，且成本递增。这种分层避免了「大炮打蚊子」——对于轻微超限，使用轻量级方案；只有在极端情况下才付出高成本。

### 3.2 模型回退的优雅降级

当主模型（如 Claude Opus）不可用时，系统会自动降级到备用模型（如 Sonnet）。这一过程涉及：
1. **孤儿消息处理**：丢弃不完整的 thinking blocks（有无效签名）
2. **状态清理**：重置 toolUseBlocks、assistantMessages
3. **签名剥离**：降级模型不支持 thinking 签名时，主动移除以避免 API 400

这种设计确保了「降级」不等于「崩溃」，系统在任何模型组合下都能保持可用。

## 四、与外部系统的边界设计

### 4.1 Bridge 模式的隔离边界

Bridge 层（CCR - Claude Code Remote）实现了本地进程与远程控制平面的边界：

```
本地 Query Loop ──▶ Bridge Client ──▶ 远程 Control Plane
      │                                      │
      └────── 消息、工具结果、状态 ────────────┘
              
              ◀──── 新请求、配置更新 ──────────
```

关键设计决策：
- **长轮询 vs WebSocket**：使用长轮询（long polling）而非 WebSocket，因为前者对代理和企业防火墙更友好
- **心跳与租约**：本地进程定期发送心跳，远程控制平面维护「活跃租约」，过期后可重新调度
- **OAuth Token 刷新**：Bridge 层负责在本地处理 Token 刷新，对上层 Query Loop 透明

### 4.2 Stop Hooks 的拦截点设计

Stop Hooks 提供了在 Query Loop 关键节点插入自定义逻辑的能力：

```
Turn 结束
   │
   ▼
Stop Hooks 执行
   ├── 阻塞错误 ──▶ 注入消息，阻止 continuation
   ├── 非阻塞错误 ──▶ 仅记录日志
   └── preventContinuation ──▶ 完全停止本轮
```

这种设计借鉴了 Web 框架的中间件（Middleware）模式，但针对 Agent 场景做了调整：
- **生成器接口**：Hook 也是 AsyncGenerator，可以流式产生进度
- **决策反馈**：Hook 可以返回结构化决策，而非简单的异常抛出
- **副作用隔离**：Hook 在独立的 subprocess 中运行，崩溃不影响主循环

## 五、反思与未来演进

### 5.1 当前架构的权衡

**优势**：
- 流式优先设计带来极致的响应速度体验
- 分层状态管理支持复杂的恢复和调试场景
- 防御性并发确保安全性不随性能提升而下降

**挑战**：
- `query.ts` 的 1700+ 行代码密度较高，状态机逻辑仍有膨胀趋势
- AsyncLocalStorage 的隐式上下文传递增加了调试难度
- 四级压缩策略的参数调优（阈值、缓冲大小）仍依赖经验

### 5.2 可能的演进方向

**更智能的压缩策略**：
当前压缩基于简单的 token 计数。未来可以引入「重要性评分」——基于注意力机制或启发式规则（如用户明确引用的内容优先保留），实现更智能的上下文修剪。

**分布式 Query Loop**：
随着 Agent 能力增强，单进程的 Query Loop 可能成为瓶颈。可以考虑将工具执行层完全分离到远程 Worker，本地仅保留编排逻辑。

**预测性执行**：
基于模型的隐状态（hidden states）预测接下来可能调用的工具，提前发起「推测性执行」，进一步降低延迟。

---

Agent Loop 的设计代表了当前 AI 编程助手领域的工程巅峰。它不是简单的「API 包装器」，而是一个完整的分布式系统运行时，在单进程内实现了流式处理、并发控制、容错恢复和企业级集成。理解其设计哲学，对于构建任何生产级 LLM 应用都具有重要参考价值。
